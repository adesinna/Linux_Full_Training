3. Hard Links and Soft Links

Inode: An inode is a data structure on a filesystem that stores all the information about a file except for its actual
name and the actual data inside the file.

- Hard Link
Suppose people are using the linux system, and A has a picture of the dept annual conference, and decides to share it
with all people using the linux system, instead of copying this picture to every user's home directory, you can create an
hard link

ln /path/target  /path/hardlink

The hard link is like any other file, but instead of having its own inode, it is pointing to the inode of the original
file. If anyone deletes his or her link, nothing happens to the file, even if the original file is deleted, the only
time the file is deleted is when there is no other hardlink pointing to that file.

However, there are drawbacks to using hard link
i. They can only be done to files
ii. They can only on the same filesystem, i.e if you have an external drive, you can not hardlink a file to your computer
file system.
iii. Make sure there is the require permission on the path where your are making the hardli

- Soft Links
Softlinks are similar to shortcuts,  it is a file that points to another file.

ln -s /path/target  /path/hardlink

When you delete the file, or change the path, the softlink breaks. It is best you use relative path for the soft link.

5. SUID, SGID and Sticky Bit

SUID (Set User ID)
This is a special permission that allows users to run an executable with the permissions of the executable's owner.
It is like giving someone your food pass to get him or her food at the lobby, even though they don’t have the pass themselves.

What it does: When set on an executable file, the program runs with the permissions of the file owner, not the user who ran it.

Common example: passwd command has SUID root, so any user can change their password even though /etc/shadow is root-owned.

ls -l

-rw-r--r-- 1 user user   31 Feb  4 18:02 shina.txt

chmod 4644 shina.txt
-rwSr--r-- 1 user user   31 Feb  4 18:02 shina.txt # this means SUID is set but no x permission

chmod 4744 shina.txt
-rwsr--r-- 1 user user   31 Feb  4 18:02 shina.txt # this means SUID is set but there is x permission


SGID (Set Group ID)

What it does:

Files: Runs with the permissions of the file’s group.

Directories: Any new files created inside inherit the directory’s group, not the user’s primary group.

ls -l

-rw-r--r-- 1 user user   31 Feb  4 18:02 shina.txt

chmod 2644 shina.txt
-rw-r--r-- 1 user user   31 Feb  4 18:02 shina.txt # this means SGID is set but no x permission

chmod 2744 shina.txt
-rwxr-Sr-- 1 user user   31 Feb  4 18:02 shina.txt # this means SGID is set but there is x permission

chmod 2714 shina.txt
-rwx--sr-- 1 user user   31 Feb  4 18:02 shina.txt


Stickybit
This protects the file from being deleted, only owner and root

chmod 1715 shina.txt

| Command     | Meaning                  | Rough time window  |
| ----------- | ------------------------ | ------------------ |
| `-mmin 10`  | ~10 minutes ago          | ~11:49:31–11:50:29 |
| `-mmin -10` | less than 10 minutes ago | ~11:50:31–12:00:00 |
| `-mmin +10` | more than 10 minutes ago | before ~11:49:30   |

find /path/ -size 512k -iname s* # this implies AND

find /path/ -size 512k -o -iname s* # -o means OR

find /path/  -not -size 512k # means NOT or use \!

find /path/ -perm 664  # Exactly this permission

find /path/ -perm -664 # At least this permission

find /path/ -perm /664 # This will show if any of the permission is there! example if user have 4 or 2, it will show

find /path/ -perm 600 file # only the owners can read and write

find /path/ -perm 100  file # only the owners can at least execute, the rest can be anything

find /path/ -not -perm -o=r  # other cannot read
